## 1. 总览：两条推理管线

两个项目的推理流程惊人地相似——都是**"编码器理解输入 → 解码器生成输出"**的经典 Encoder-Decoder 模式：

```
┌──────────────────────────────────────────────┐
│              通用推理管线                      │
│                                              │
│  原始输入 → [预处理] → [编码器] → [投影]       │
│                                    ↓         │
│           [构建 Prompt] → [Prefill] → [Decode 循环] → 输出文字  │
└──────────────────────────────────────────────┘

ds-ocr:   图像 → SAM → DeepEncoder → Projector → MoE Decoder → OCR文字
qwen-asr: 音频 → Mel → Conv+Transformer → Proj → LLM Decoder → 转录文字
```

## 2. ds-ocr 全流程：图像 → 文字

### 2.0 入口：`ds_recognize()` → `ds_recognize_image()`

```c
char *ds_recognize(ds_ctx_t *ctx, const char *image_path) {
    ds_image_t *img = ds_image_load(image_path);  /* 加载图像 */
    return ds_recognize_image(ctx, img->pixels, img->width, img->height, img->channels);
}
```

用户调用 `ds_recognize()`，传入图像路径，返回识别出的文字。

### 2.1 Step 1: 图像预处理

**单图模式**（小图像 ≤ 768×768）：
- Pad 到 1024×1024（填充值 127）

**多裁剪模式**（大图像 > 768×768）：
- 生成 N 个 768×768 的局部裁剪（local crops）
- 1 个 1024×1024 的全局缩略图（global view）

```c
if (width > 768 || height > 768) {
    /* 大图像：多裁剪 */
    crops = ds_dynamic_preprocess(&img, 768, 1, 12, 0, &n_crops);
    /* n_crops 通常 = 6 (6个768×768局部裁剪) */
    /* 全局图: 原图 pad 到 1024×1024 */
} else {
    /* 小图像：直接 pad 到 1024×1024 */
    global_img = ds_image_pad(&img, 1024, 127);
}
```

### 2.2 Step 2: SAM 视觉编码

每个裁剪/全局图通过 **SAM ViT-B** 提取视觉特征：

```
图像 [3, 1024, 1024]
    ↓ Patch Embed: Conv2d(3→768, k=16, s=16)
    ↓ [768, 64, 64] = 4096 patches
    ↓ + 位置编码 [577, 768]
    ↓ 12 层 Transformer (window attention + global attention)
    ↓ SAM Neck: Conv2d+LN → Conv2d+LN → 下采样
    ↓ SAM 特征: [n_tokens, 896]  (V2) 或 [n_tokens, 1024] (V1)
```

**关键细节**：
- SAM 使用**窗口注意力**（window_size=14）+ **全局注意力**（在 layer 2,5,8,11）
- 窗口注意力需要**相对位置编码**（rel_pos_h, rel_pos_w）
- 全局注意力时，所有 patch 互相可见

```c
/* SAM 前向传播 */
float *ds_sam_forward_image(ds_ctx_t *ctx, const ds_image_t *img,
                             int *out_n_tokens, float **out_patch_embeds);
/* 输出: SAM 特征 [n_tokens × sam_ds2_dim] */
/* 1024×1024 → 64/4 × 64/4 = 256 tokens */
/* 768×768 → 48/4 × 48/4 = 144 tokens */
```

**多裁剪并行**：
```c
/* 6 个局部裁剪并行编码 */
for (int i = 0; i < n_crops; i++)
    pthread_create(&threads[i], NULL, crop_worker, &tasks[i]);
for (int i = 0; i < n_crops; i++)
    pthread_join(threads[i], NULL);
```

### 2.3 Step 3: DeepEncoder V2 编码

SAM 特征通过 **DeepEncoder V2**（Qwen2-0.5B 架构）进一步编码：

```
SAM 特征 [256, 896]
    ↓ + 因果流查询 [256, 896]  ← 可学习的查询向量
    ↓ 24 层 Transformer (混合注意力)
    │   视觉 token: 双向注意力
    │   因果查询: 因果注意力
    ↓ 最终 RMSNorm
    ↓ 输出 [256, 896]
```

```c
float *ds_encoder_forward_v2(ds_ctx_t *ctx, const float *visual_tokens,
                               int n_tokens, int *out_seq_len,
                               int n_causal_queries, const float *causal_queries);
```

> **因果流查询**是 DeepSeek-OCR V2 的创新：256 个可学习的查询向量通过因果注意力逐个"收集"视觉信息，输出直接传给解码器。

### 2.4 Step 4: Projector 投影

将编码器输出投影到解码器的隐藏维度：

```
DeepEncoder 输出 [256, 896]
    ↓ Linear(896 → 1280)
    ↓ 投影后 [256, 1280]
```

```c
/* Projector: 简单的线性层 */
ds_linear(encoder_out, encoder_input, projector.weight, projector.bias,
          seq_len, proj_input_dim, dec_hidden);
/* proj_input_dim = 896 (V2) 或 2048 (V1: CLIP+SAM 拼接) */
```

### 2.5 Step 5: 构建 Decoder 输入序列

解码器的输入序列由以下部分组成：

**V2 格式**：
```
[BOS] + [编码器输出 tokens] + ["\nFree OCR."] 
```

**多裁剪时的 token 布局**：
```
[BOS(1)] + [local_crops(864) + global(256) + view_sep(1)] + [text(4)] = 1126 tokens
```

**代码实现**（`ds_ocr.c`）：

```c
/* 1. BOS token (id=0) */
EMBED_TOKEN(DS_TOKEN_BOS);

/* 2. 编码器输出 tokens (直接复制 float 向量) */
memcpy(input_embeds + img_pos * hidden, enc, n_copy * hidden * sizeof(float));

/* 3. 文本 token: "\nFree OCR." → [201, 21431, 126041, 16] */
for (int t = 0; t < n_text_after; t++) EMBED_TOKEN(text_after_ids[t]);
```

> **重要**：编码器输出是**连续的 float 向量**，不经过词嵌入查表。它直接作为 decoder 的输入嵌入（input_embeds），这和文本 token 通过查表获取嵌入不同。

### 2.6 Step 6: Prefill（预填充）

将整个输入序列一次性送入解码器：

```c
/* Prefill: 批量处理所有 prompt tokens */
ds_decoder_prefill(ctx, input_embeds, prefix_len);

/* Prefill 内部：
 * 1. 对每个 token: 嵌入 → RMSNorm → QKV → RoPE → 注意力 → FFN/MoE
 * 2. 所有 token 的 K、V 存入 KV 缓存
 * 3. 从最后一个位置得到第一个预测 token
 */
```

**Prefill 的输出**：最后一个位置的 logits → argmax → 第一个生成的 token ID。

### 2.7 Step 7: Decode（自回归生成）

逐个生成 token，直到遇到 EOS 或达到最大长度：

```c
for (int step = 0; step < ctx->max_new_tokens; step++) {
    /* 1. 当前 token 的嵌入 */
    const uint16_t *emb = tok_embeddings_bf16 + token * hidden;
    /* BF16 → F32 */
    for (int i = 0; i < hidden; i++) {
        uint32_t f32_bits = ((uint32_t)emb[i]) << 16;
        memcpy(&dec_input[i], &f32_bits, sizeof(float));
    }
    
    /* 2. 单 token 前向传播 */
    int token = ds_decoder_forward(ctx, dec_input);
    
    /* 3. 检查 EOS */
    if (token == DS_TOKEN_EOS) break;
    
    /* 4. 解码 token → 文字 */
    const char *piece = ds_tokenizer_decode(tokenizer, token);
    /* 拼接到输出 */
    
    /* 5. 更新 token 历史（用于重复惩罚） */
    ctx->token_history[ctx->token_history_len++] = token;
}
```

### 2.8 完整数据流图

```

              │
    ┌─────────┴──────────┐
    │  Step 1: 图像预处理  │
    │  pad/resize → 1024² │
```

## 3. qwen-asr 全流程：音频 → 文字

### 3.0 入口：`qwen_transcribe()`

```c
char *qwen_transcribe(qwen_ctx_t *ctx, const char *wav_path) {
    float *samples = qwen_audio_load_wav(wav_path, &n_samples);
    return qwen_transcribe_audio(ctx, samples, n_samples);
}
```

### 3.1 Step 1: 音频预处理 → Mel 频谱图

```c
float *mel = qwen_mel_spectrogram(samples, n_samples, &mel_frames);
/* 输出: [128, mel_frames] — 128个Mel频率bin × T帧 */
```

Mel 频谱图参数：
- 采样率：16000 Hz
- 窗口大小：400（25ms）
- 跳跃长度：160（10ms）
- Mel bins：128

### 3.2 Step 2: 音频编码器

```
Mel 频谱图 [128, T]
    ↓ Conv2D stem: 3层 Conv2d(3×3, stride=2) + GELU
    ↓ [480, 16, T/8] → reshape → [T/8, 7680]
    ↓ Linear(7680 → d_model) + 正弦位置编码
    ↓ 24 层 Transformer (窗口化双向注意力)
    ↓ 最终 LayerNorm
    ↓ proj1 + GELU + proj2
    ↓ 输出 [T/8, output_dim]
```

**分块处理**：音频被分成多个 chunk，每个 chunk 独立编码：

```c
int chunk_size = 100;  /* 100 个 Mel 帧 ≈ 1 秒音频 */
int n_chunks = (mel_frames + chunk_size - 1) / chunk_size;
```

### 3.3 Step 3: 构建 Decoder 输入

qwen-asr 的 prompt 格式：
```
[<audio_start>] + [audio_tokens(N)] + [<audio_end>] + [<asr_text>] + [system_prompt?]
```

### 3.4 Step 4: Prefill + Decode

```c
qwen_decoder_prefill(ctx, input_embeds, prefix_len);

for (int step = 0; step < max_new_tokens; step++) {
    int token = qwen_decoder_forward(ctx, dec_input);
    if (token == QWEN_TOKEN_ENDOFTEXT) break;
    const char *piece = qwen_tokenizer_decode(tokenizer, token);
}
```

## 4. 两个流程的对比与共性

### 4.1 共性模式

两个项目遵循完全相同的推理模式：

```
1. 原始输入 → 预处理
2. 预处理输出 → 编码器（理解输入）
3. 编码器输出 → 投影（维度对齐）
4. 投影输出 + 特殊 token → 构建 Prompt
5. Prompt → Prefill（批量处理）
6. Prefill 输出 → Decode 循环（逐 token 生成）
7. 生成的 token → 解码为文字
```

### 4.2 关键差异

| 维度 | ds-ocr | qwen-asr |
|------|--------|----------|
| **输入模态** | 2D 图像 | 1D 音频频谱 |
| **预处理** | Resize/Pad + 多裁剪 | Mel 频谱图 + 分块 |
| **编码器** | SAM ViT (视觉) + DeepEncoder (深化) | Conv2D stem + Transformer (音频) |
| **投影** | Linear(896→1280) | Linear(d_model→output_dim) |
| **解码器** | MoE (3B总参, 570M激活) | Dense (1.7B或0.6B全部激活) |
| **注意力** | MHA (解码器) + 混合注意力 (编码器) | GQA (解码器) + 窗口注意力 (编码器) |
| **特殊设计** | 因果流查询 + 共享专家 | 窗口化编码 + 音频分块 |
| **Prompt 格式** | BOS + img_tokens + "Free OCR." | audio_start + tokens + audio_end + asr_text |

### 4.3 编码器的设计哲学差异

| | SAM ViT (ds-ocr) | 音频 Transformer (qwen-asr) |
|--|-------------------|---------------------------|
| **位置编码** | 可学习 2D 位置 + 相对位置 | 正弦 1D 位置编码 |
| **注意力方式** | 窗口 + 全局交替 | 窗口化双向 |
| **归一化** | LayerNorm (有 bias) | LayerNorm (有 bias) |
| **FFN** | GELU FFN | GELU FFN |
| **额外处理** | Neck + Downsample | Conv2D stem 前置 |
| **输出** | 空间特征 → 压缩为 token | 时序特征 → 直接投影 |

**核心区别**：
- SAM ViT 先用 Patch Embed 把图像切成 patch，再用 Transformer 处理，最后通过 Neck+Downsample 压缩
- 音频编码器先用 Conv2D stem 压缩频谱，再直接用 Transformer 处理已压缩的特征

> 两种方式殊途同归：**先用卷积压缩，再用 Transformer 建模**。只是压缩发生在不同位置——SAM 在后端（Neck），音频编码器在前端（stem）。

## 5. 性能分析与优化空间

### 5.1 典型耗时分布

**ds-ocr**（1024×1024 图像，Mac M 系列）：

| 阶段 | 耗时占比 | 说明 |
|------|---------|------|
| SAM 编码 | ~30% | 12层 ViT，计算密集 |
| DeepEncoder | ~20% | 24层 Transformer |
| Prefill | ~15% | 1000+ tokens 批量处理 |
| Decode | ~35% | 逐 token 生成，内存带宽瓶颈 |

**qwen-asr**（30 秒音频）：

| 阶段 | 耗时占比 | 说明 |
|------|---------|------|
| Mel 频谱图 | <1% | 简单的 STFT |
| 音频编码器 | ~20% | Conv stem + 24 层 Transformer |
| Prefill | ~15% | 音频 tokens 批量处理 |
| Decode | ~65% | 逐 token 生成，主要瓶颈 |

### 5.2 解码器的内存带宽瓶颈

单 token 解码时，每次需要加载：
- QKV 权重：3 × 1280 × 1280 × 2 = ~9.4MB (BF16)
- 注意力：KV 缓存读取 ~1280 × seq_len × 4 bytes
- MoE 专家：6 × (3 × 1280 × 896 × 2) = ~40MB (BF16)
- 共享专家：(3 × 1280 × 1792 × 2) = ~13MB (BF16)

总计每次 decode step 需读取 **~63MB 权重**。以 M2 Max 的带宽 400GB/s 计算，理论极限约 6 token/s。

> **这就是为什么 MoE 的 decode 更慢**：6 个路由专家 + 2 个共享专家需要读取更多权重，相比稠密 FFN 的单个权重矩阵。

### 5.3 优化方向

| 优化 | 原理 | 预期收益 |
|------|------|---------|
| **权重量化(INT4/INT8)** | 减少内存带宽需求 | 2-4× decode 加速 |
| **Speculative Decoding** | 小模型预测多 token，大模型验证 | 2-3× decode 加速 |
| **Expert权重预取** | 根据路由预测提前加载专家权重 | 减少延迟 |
| **FlashAttention** | 减少注意力的内存读写 | prefill 加速 |
| **KV 缓存量化** | 减少长序列的内存占用 | 长文本场景 |
| **Batched Decode** | 多请求批量 decode | 吞吐量提升 |

### 5.4 ds-ocr vs qwen-asr 的 decode 速度差异

| | ds-ocr (MoE) | qwen-asr (Dense) |
|--|--------------|------------------|
| **每步权重读取** | ~63MB (6专家+共享) | ~15MB (1个FFN) |
| **decode 速度** | 较慢 | 较快 |
| **知识容量** | 大 (3B 参数) | 中 (1.7B 参数) |

> MoE 的 decode 慢是**结构性的**——每次要读取更多权重。但换来的是更大的模型容量和更好的输出质量。这是**质量 vs 速度**的经典权衡。

## 小结

现在你已经完整理解了两个项目从输入到输出的全链路：

1. **ds-ocr**：图像 → SAM → DeepEncoder → Projector → MoE Decoder → 文字
2. **qwen-asr**：音频 → Mel → Conv+Transformer → Proj → Dense Decoder → 文字
3. **共性模式**：预处理 → 编码 → 投影 → 构建 Prompt → Prefill → Decode → 解码
4. **性能瓶颈**：Decode 阶段的内存带宽，MoE 尤为突出
