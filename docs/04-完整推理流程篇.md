## 1. 总览：三条推理管线 + 对照

ds-ocr 三个版本（V1、V2、V3）遵循相同的**"编码器理解图像 → 解码器生成文字"**范式，但编码器架构和解码器注意力机制各不相同：

```
┌──────────────────────────────────────────────────────────────┐
│                    通用推理管线                                │
│                                                              │
│  原始图像 → [预处理] → [视觉编码器] → [投影]                  │
│                                    ↓                         │
│           [构建 Prompt] → [Prefill] → [Decode 循环] → OCR文字 │
└──────────────────────────────────────────────────────────────┘

ds-ocr V1:  图像 → SAM ViT-B → CLIP ViT-L/14 → Concat → Projector → MoE Decoder → OCR文字
ds-ocr V2:  图像 → 多裁剪+SAM → DeepEncoder V2 → Projector → MoE Decoder → OCR文字
ds-ocr V3:  图像 → SAM ViT-B → CLIP ViT-L/14 → Concat → Projector → MoE Decoder(R-SWA) → OCR文字
```

**三版核心差异一览**：

| | V1 (DeepSeek-OCR) | V2 (DeepEncoder V2) | V3 (Unlimited-OCR) |
|--|-------------------|---------------------|---------------------|
| **编码器** | SAM + CLIP 双编码器 | SAM + Qwen2-0.5B | SAM + CLIP 双编码器 |
| **特征融合** | Concat(SAM, CLIP) → 2048-dim | 因果流查询 → 896-dim | Concat(SAM, CLIP) → 2048-dim |
| **解码注意力** | 标准因果注意力 | 标准因果注意力 | R-SWA（参考滑动窗口） |
| **图像输入** | 1024×1024 拉伸 | 动态多裁剪 | 1024×1024 拉伸 |
| **图像 token** | 273（含 newline + view_sep） | 256+（含 view_sep） | 273（含 newline + view_sep） |
| **特殊设计** | CLIP 接收 SAM 特征 | 因果流查询 | R-SWA + det/ref 标签过滤 |
| **适用场景** | 通用 OCR | 高分辨率多裁剪 | 长文档无限制解析 |

> **qwen-asr 对照**：音频 → Mel → Conv+Transformer → Proj → Dense Decoder → 转录文字。两个项目共享相同的解码骨架（RMSNorm → 注意力 → MoE/FFN）和内核实现。

### 版本自动检测

ds-ocr 通过读取 `config.json` 自动检测模型版本（`ds_ocr.c: detect_model_version()`）：

```c
/* 检测顺序：V3 → V2 → V1 */
if (strstr(buf, "sliding_window_size") || strstr(buf, "unlimited-ocr"))
    return 3;  /* V3: 有 sliding_window_size 字段 */
if (strstr(buf, "DeepEncoderV2") || strstr(buf, "causal_flow"))
    return 2;  /* V2: 有 DeepEncoderV2 或 causal_flow */
return 1;       /* V1: 默认 */
```

## 2. ds-ocr 全流程：图像 → 文字

### 2.0 入口：`ds_recognize()` → `ds_recognize_image()`

```c
char *ds_recognize(ds_ctx_t *ctx, const char *image_path) {
    ds_image_t *img = ds_image_load(image_path);  /* 加载图像 */
    return ds_recognize_image(ctx, img->pixels, img->width, img->height, img->channels);
}
```

用户调用 `ds_recognize()`，传入图像路径，返回识别出的文字。

### 2.1 Step 1: 图像预处理

**V2 多裁剪模式**（大图像 > 768×768）：
- 生成 N 个 768×768 的局部裁剪（local crops）
- 1 个 1024×1024 的全局缩略图（global view）

```c
if (width > 768 || height > 768) {
    /* V2 大图像：多裁剪 */
    crops = ds_dynamic_preprocess(&img, 768, 1, 12, 0, &n_crops);
    /* n_crops 通常 = 6 (6个768×768局部裁剪) */
    /* 全局图: 原图 pad 到 1024×1024 */
} else {
    /* 小图像：直接 pad 到 1024×1024 */
    global_img = ds_image_pad(&img, 1024, 127);
}
```

**V1/V3 单图模式**：
- 统一拉伸到 1024×1024（stretch resize，不做动态多裁剪）
- 输入分辨率固定，确保 CLIP 接收 256 个 token

> **关键差异**：V2 使用动态多裁剪以处理高分辨率图像，V1/V3 使用固定 1024×1024 拉伸。这是因为 V1/V3 的 CLIP 编码器需要固定的空间结构（16×16 grid）来与 SAM 特征对齐。

### 2.2 Step 2: SAM 视觉编码（三版共用）

每个裁剪/全局图通过 **SAM ViT-B** 提取视觉特征：

```
图像 [3, 1024, 1024]
    ↓ Patch Embed: Conv2d(3→768, k=16, s=16)
    ↓ [768, 64, 64] = 4096 patches
    ↓ + 位置编码 [577, 768]
    ↓ 12 层 Transformer (window attention + global attention)
    ↓ SAM Neck: Conv2d+LN → Conv2d+LN → 下采样
    ↓ SAM 特征
```

**输出维度因版本不同**：
- **V1/V3**：SAM Neck 输出 [n_tokens, **1024**]（`cfg->sam_ds2_dim = 1024`）
- **V2**：SAM Neck 输出 [n_tokens, **896**]（`cfg->sam_ds2_dim = 896`）

**关键细节**：
- SAM 使用**窗口注意力**（window_size=14）+ **全局注意力**（在 layer 2,5,8,11）
- 窗口注意力需要**相对位置编码**（rel_pos_h, rel_pos_w）
- 全局注意力时，所有 patch 互相可见

```c
/* SAM 前向传播 */
float *ds_sam_forward_image(ds_ctx_t *ctx, const ds_image_t *img,
                             int *out_n_tokens, float **out_patch_embeds);
/* 输出: SAM 特征 [n_tokens × sam_ds2_dim] */
/* 1024×1024 → 64/4 × 64/4 = 256 tokens */
/* 768×768 → 48/4 × 48/4 = 144 tokens */
```

**V2 多裁剪并行**：
```c
/* 6 个局部裁剪并行编码 */
for (int i = 0; i < n_crops; i++)
    pthread_create(&threads[i], NULL, crop_worker, &tasks[i]);
for (int i = 0; i < n_crops; i++)
    pthread_join(threads[i], NULL);
```

> **从 Step 3 开始，三条路径分叉**：V2 走 DeepEncoder V2（2.3），V1/V3 走 CLIP 双编码器（2.4/2.5）。

### 2.3 V2 编码管线：DeepEncoder V2

SAM 特征通过 **DeepEncoder V2**（Qwen2-0.5B 架构）进一步编码：

```
SAM 特征 [256, 896]
    ↓ + 因果流查询 [256, 896]  ← 可学习的查询向量
    ↓ 24 层 Transformer (混合注意力)
    │   视觉 token: 双向注意力
    │   因果查询: 因果注意力
    ↓ 最终 RMSNorm
    ↓ 输出 [256, 896]
```

```c
float *ds_encoder_forward_v2(ds_ctx_t *ctx, const float *visual_tokens,
                               int n_tokens, int *out_seq_len,
                               int n_causal_queries, const float *causal_queries);
```

> **因果流查询**是 DeepSeek-OCR V2 的创新：256 个可学习的查询向量通过因果注意力逐个"收集"视觉信息，输出直接传给解码器。

### 2.4 V1 编码管线：SAM + CLIP 双编码器

V1 采用 **双编码器架构**——SAM 提取底层视觉特征，CLIP 提供高层语义理解，两者特征拼接后送入解码器。

#### 2.4.1 架构总览

```
图像 [3, 1024, 1024]
    ↓ SAM ViT-B (12 层, window attention)
    ↓ SAM Neck: 256→512→1024 channels
    ↓ SAM 特征 [256, 1024]  ──────────────┐
    │                                     │ Concat (feature dim)
    ↓ CLIP ViT-L/14 (24 层, 16 heads)    │
    ↓ (SAM 特征作为 CLIP patch_embeds)    │
    ↓ CLIP 输出 [256, 1024]  ────────────┘
    ↓
    ↓ 拼接特征 [256, 2048]
    ↓ Projector: Linear(2048→1280)
    ↓ + image_newline + view_seperator
    ↓ 最终: 273 tokens [273, 1280]
```

#### 2.4.2 CLIP 如何接收 SAM 特征

这是 V1/V3 最独特的设计：**CLIP 绕过自身的 Conv2d patch_embed，直接接收 SAM 特征作为输入**。

```c
/* ds_deep_encoder.c: ds_clip_encoder_forward() */

/* Step 1: 用 SAM 特征直接作为 CLIP 的 patch_embeds
 * CLIP 的 patch_embedding Conv2d [1024, 3, 14, 14] 完全不使用！
 * SAM 的 16× 压缩已经产生了 256 个 token，CLIP 直接处理它们。
 */
clip_n_patches = n_sam_tokens;  /* 256 for 1024×1024 input */
clip_tokens = (float *)malloc(clip_n_patches * clip_dim * sizeof(float));
memcpy(clip_tokens, sam_features, clip_n_patches * clip_dim * sizeof(float));
```

**为什么要绕过 patch_embed？**
- CLIP 原始设计是对 RGB 图像做 14×14 的 patch 切分，每个 patch 投影为 1024 维向量
- 但 V1/V3 中，SAM 已经完成了类似的"patch 化 + 投影"工作，且更精细（包含多尺度信息）
- 让 CLIP 直接处理 SAM 特征，相当于"二次精炼"——SAM 提取底层特征，CLIP 提供高层语义

#### 2.4.3 CLIP 内部处理

```
SAM 特征 [256, 1024]
    ↓ 添加 CLS token → [257, 1024]
    ↓ + 位置编码 (get_abs_pos, 可能需要插值)
    ↓ Pre-LayerNorm
    ↓ 24 层 Transformer (标准 ViT: LN→MHA→残差→LN→MLP→残差)
    ↓ 移除 CLS token → [256, 1024]
```

**位置编码插值**：
CLIP 原始位置编码为 257 个（1 CLS + 16×16 = 256 patches）。当 `clip_n_patches = 256` 时完美匹配，无需插值。对于非标准尺寸，需要双线性插值：

```c
/* ds_deep_encoder.c: 位置编码插值 */
if (total_len <= 257) {
    /* 直接复制 — 257 个位置编码正好够用 */
    for (int i = 0; i < total_len; i++)
        for (int d = 0; d < clip_dim; d++)
            x[i * clip_dim + d] += clip->position_embedding[i * clip_dim + d];
} else {
    /* 双线性插值: [16×16] → [tgt_h × tgt_w] */
    int src_size = 16;  /* CLIP 原始空间尺寸 */
    int tgt_size = (int)sqrtf((float)clip_n_patches);
    /* ... bicubic interpolation code ... */
}
```

#### 2.4.4 特征拼接 + Projector

CLIP 和 SAM 的特征沿特征维度拼接：

```c
/* ds_deep_encoder.c: 特征拼接 */
/* V1/V3: CLIP 和 SAM 有相同 token 数量 (256 each for 1024×1024) */
/* Concat: [256, 1024] + [256, 1024] → [256, 2048] along feature dim */
concat_len = clip_output_len;
proj_in_dim = clip_dim + cfg->sam_ds2_dim; /* 1024 + 1024 = 2048 */
concat_features = (float *)malloc(concat_len * proj_in_dim * sizeof(float));
for (int t = 0; t < concat_len; t++) {
    memcpy(concat_features + t * proj_in_dim,
           clip_output + t * clip_dim, clip_dim * sizeof(float));
    memcpy(concat_features + t * proj_in_dim + clip_dim,
           sam_features + t * cfg->sam_ds2_dim, cfg->sam_ds2_dim * sizeof(float));
}

/* Projector: Linear(2048 → 1280) */
ds_linear(projected, concat_features, ctx->projector.weight, ctx->projector.bias,
          concat_len, proj_in_dim, dec_hidden);
```

> **对比 V2**：V2 的 Projector 输入维度是 896（仅 DeepEncoder 输出），V1/V3 的 Projector 输入维度是 2048（CLIP 1024 + SAM 1024 拼接），所以 V1/V3 的 Projector 权重更大。

#### 2.4.5 image_newline + view_seperator 布局

V1/V3 的 273 个图像 token 有特殊的空间布局：

```
16 rows × (16 patches + 1 image_newline) + 1 view_seperator = 273 tokens

Row 0:  [patch_0, patch_1, ..., patch_15, image_newline]   ← 17 tokens
Row 1:  [patch_16, patch_17, ..., patch_31, image_newline]  ← 17 tokens
...
Row 15: [patch_240, ..., patch_255, image_newline]          ← 17 tokens
Last:   [view_seperator]                                     ← 1 token
Total:  16 × 17 + 1 = 273 tokens
```

```c
/* ds_deep_encoder.c: 构建 2D grid 布局 */
int grid_h = (int)sqrtf((float)concat_len); /* 16 */
int grid_w = concat_len / grid_h;           /* 16 */
float *image_newline = ctx->vis_tokenizer.image_newline;   /* [1280] 可学习 */
float *view_sep = ctx->vis_tokenizer.view_seperator;       /* [1280] 可学习 */

/* Build: [row0_patches, newline, row1_patches, newline, ..., view_sep] */
int final_len = grid_h * (grid_w + 1) + 1; /* 16*17+1 = 273 */
for (int row = 0; row < grid_h; row++) {
    memcpy(final_out + dst * dec_hidden,
           projected + row * grid_w * dec_hidden,
           grid_w * dec_hidden * sizeof(float));
    dst += grid_w;
    memcpy(final_out + dst * dec_hidden, image_newline, dec_hidden * sizeof(float));
    dst += 1;
}
memcpy(final_out + dst * dec_hidden, view_sep, dec_hidden * sizeof(float));
```

> **为什么需要 image_newline？** 它相当于"行分隔符"，告诉解码器图像空间结构的行边界。这类似于文本中的换行符，帮助解码器理解图像 token 的二维排列。view_seperator 则标记整个图像 token 序列的结束。

#### 2.4.6 V1 的 Prompt 格式

V1 的解码器输入序列：

```
[BOS] + [img_start] + [273_图像 tokens] + [img_end] + ["\nFree OCR."]
```

- 273 图像 token = 16 行 × (16 SAM + 1 newline) + 1 view_seperator
- `image_token_id`：V1 使用 `DS_TOKEN_IMAGE_START` 作为图像占位符
- Prompt 文本：`"\nFree OCR."` → token IDs [201, 21431, 126041, 16]
- **max_position_embeddings**：8192

### 2.5 V3 编码管线：SAM + CLIP 双编码器 + R-SWA 解码

V
