## 1. 基线性能与瓶颈分析

### 1.1 优化前后的全景对比

| 阶段 | 总耗时 | SAM+Encoder | Prefill | Decode | 测试平台 |
|------|--------|-------------|---------|--------|---------|
| **v0.5（基线）** | ~97s | ~50s | ~30s | ~19s | M2 Pro |
| **v0.9（当前）** | ~15s | ~9s | ~0.93s | ~5.1s | M2 Pro |
| **加速比** | 6.5× | 5.5× | 32× | 3.7× | — |
| **Python PyTorch** | ~736s | — | — | — | 同平台 |

> **对比意义**：纯 C 推理引擎 v0.5 就已比 Python PyTorch 快 7.5×，而 v0.9 进一步将优势扩大到 **49×**。这并非魔法——每一倍加速都来自对硬件特性的深度理解。

### 1.2 瓶颈定位方法论

优化的第一步不是写代码，而是**测量**。盲目优化往往耗时巨大却收效甚微。

**ds-ocr 的性能剖析流程**：

```
Step 1: 粗粒度测量 → 哪个阶段最慢？
         ┌──────────────────────────────────────────┐
         │  Encoding: 50s  │ Prefill: 30s │ Decode: 19s │
         │     ← 最慢 ─────┘              │            │
         └──────────────────────────────────────────┘

Step 2: 细粒度 Profiler → 阶段内部哪个操作最慢？
         ┌─── Decode 内部 ────────────────────────────┐
         │ QKV: 4.6% │ Attn: 9.9% │ Proj: 3.2%       │
         │ MLP/MoE: 73.7%  ← 绝对瓶颈               │
         │ LM head: 6.9% │ Sampling: 1.7%             │
         └────────────────────────────────────────────┘

Step 3: 针对瓶颈设计优化方案
```

ds-ocr 内置了逐层 Profiler，通过 `-p` 参数启用：

```bash
./ds_ocr -d model_dir -i image.png -p
```

输出类似：

```

  QKV proj:     241.6 ms (  4.6%)
  Attention:    524.2 ms (  9.9%)

  MLP/MoE:     3890.3 ms ( 73.7%)  ← 主要瓶颈
  LM head:      365.8 ms (  6.9%)

  Per-layer breakdown (ms/step avg):
  Layer    QKV    Attn    Proj     MLP   Total
  0       0.893   1.854   0.596  13.421  16.764

  ...
```

### 1.3 Amdahl 定律的实战指导

**Amdahl 定律**：加速比 = 1 / [(1 - p) + p/s]

其中 p 是被优化部分的比例，s 是该部分的加速比。

以 Decode 阶段为例：
- MLP/MoE 占 73.7%，若优化到 2× 速：总加速 = 1 / [0.263 + 0.737/2] = **1.52×**
- LM head 占 6.9%，若优化到 10× 速：总加速 = 1 / [0.931 + 0.069/10] = **1.07×**

> **结论**：永远优先优化占比最大的部分。MLP/MoE 的 2× 加速远比 LM head 的 10× 加速更有价值。

## 2. Encoding 优化：50s → 9s（5.5×）

### 2.1 问题：V2 的多裁剪串行编码

V2 模型使用动态多裁剪策略处理高分辨率图像：

```
原始图像（如 2000×3000）
  ├── 全局裁剪（1024×1024）→ SAM → DeepEncoder → 256 tokens
  ├── 局部裁剪 1（448×448） → SAM → DeepEncoder → 56 tokens
  ├── 局部裁剪 2（448×448） → SAM → DeepEncoder → 56 tokens
  ├── 局部裁剪 3（448×448） → SAM → DeepEncoder → 56 tokens
  └── 局部裁剪 4（448×448） → SAM → DeepEncoder → 56 tokens
```

**v0.5 的做法**：完全串行处理，先全局后逐个局部：

```c
/* v0.5: 串行 —— 总时间 = global + local1 + local2 + local3 + local4 ≈ 50s */
float *global_tokens = encode_global(ctx, global_crop);   // ~12s
float *local1 = encode_local(ctx, crops[0]);              // ~9.5s
float *local2 = encode_local(ctx, crops[1]);              // ~9.5s
float *local3 = encode_local(ctx, crops[2]);              // ~9.5s
float *local4 = encode_local(ctx, crops[3]);              // ~9.5s
// 总计 ~50s
```

### 2.2 优化：并行 N+1 Workers

**核心洞察**：每个裁剪的编码完全独立（无数据依赖），天然可并行。

**v0.9 的做法**：同时启动 N+1 个线程（N 个局部 + 1 个全局），等待全部完成：

```c
/* v0.9: 并行 —— 总时间 = max(global, max(local)) ≈ 12s → 实测 9s */
int n_parallel = n_crops + 1;   // 4 local + 1 global = 5 workers
crop_task_t *tasks = calloc(n_parallel, sizeof(crop_task_t));
pthread_t *threads = calloc(n_parallel, sizeof(pthread_t));

/* 同时启动所有局部裁剪编码 */
for (int i = 0; i < n_crops; i++) {
    tasks[i].ctx = ctx;
    tasks[i].crop = crops[i];
    tasks[i].tokens_per_crop = tokens_per_crop;
    pthread_create(&threads[i], NULL, crop_worker, &tasks[i]);
}

/* 同时启动全局裁剪编码 */
int gi = n_crops;
tasks[gi].ctx = ctx;
tasks[gi].crop = global_img;
tasks[gi].tokens_per_crop = 256;
pthread_create(&threads[gi], NULL, crop_worker_global, &tasks[gi]);

/* 等待所有任务完成 */
for (int i = 0; i < n_parallel; i++)
    pthread_join(threads[i], NULL);
```

每个 worker 内部执行相同的流水线：

```c
static void *crop_worker(void *arg) {
    crop_task_t *t = (crop_task_t *)arg;
    t->sam_tokens = ds_sam_forward_image(t->ctx, t->crop,
                                          &t->n_sam_tokens, NULL);
    t->enc_tokens = ds_encoder_forward_v2(t->ctx, t->sam_tokens,
                                           t->n_sam_tokens,
                                           &t->n_enc_tokens,
                                           t->tokens_per_crop, ...);
    return NULL;
}
```

### 2.3 性能模型

```
串行时间 = T_global + N × T_local = 12 + 4 × 9.5 = 50s
并行时间 = max(T_global, max(T_local)) = max(12, 9.5) = 12s（理想值）
实测并行 = 9s（M2 Pro 性能核/能效核调度优化）
```

> **为什么实测比理想值更短？** M2 Pro 有 6 个性能核 + 4 个能效核。5 个 worker 分别调度到不同核心，能效核处理较小的局部裁剪（448×448），性能核处理全局裁剪（1024×1024），实际负载比理想的 max 值更均衡。

### 2.4 V1/V3 为何不需要此优化？

V1 和 V3 使用 1024×1024 单图输入，没有多裁剪，编码只需 ~6.4s，已不是瓶颈。

## 3. Prefill 优化：30s → 0.93s（32×）

### 3.1 问题：逐 Token 的 BF16 矩阵向量乘

Prefill 阶段需要处理整个输入序列（~860 个 token），对每个 token 执行 MoE 前向传播。

**v0.5 的做法**：与 Decode 相同——对每个 token 独立执行 BF16 matvec：

```c
/* v0.5: 860 tokens × (gate + up + down) × 6 experts = ~15,000 次独立 BF16 matvec
 * 每次 matvec: [1, 896] × [896, 1280] → 单向量乘法，BLAS 无法发挥 */
for (int s = 0; s < seq_len; s++) {
    float *x_s = x + s * hidden;
    for (int k = 0; k < top_k; k++) {
        int expert_id = top_indices[s * top_k + k];
        ds_bf16_matvec(gate_buf, x_s, gate_weight, hidden, moe_inter);
        ds_bf16_matvec(up_buf, x_s, up_weight, hidden, moe_inter);
        ds_swiglu(swiglu_buf, gate_buf, up_buf, moe_inter);
        ds_bf16_matvec(out_buf, swiglu_buf, down_weight, moe_inter, hidden);
    }
}
```

**为什么慢？** BLAS 的 sgemm 对**大批量**操作高度优化，而单向量 matvec 根本无法利用：
- CPU 的 SIMD 单元未饱和（NEON/AVX 只用了几分之一）
- 缓存命中率低（每次只读一行权重，下次可能已驱逐）
- 线程池调度开销 >> 实际计算

### 3.2 优化一：共享专家批量 sgemm

**核心洞察**：共享专家（2 个）对所有 token 都激活，输入是 `[seq_len, hidden]` 的完整矩阵。

```c
/* v0.9: 共享专家 —— 一次 sgemm 处理所有 token */
ds_linear_nobias_bf16(s_gate, x_norm, layer->shared_gate_weight_bf16,
                       seq_len, hidden, shared_inter);  // [860, 896] × [896, 2560]
ds_linear_nobias_bf16(s_up, x_norm, layer->shared_up_weight_bf16,
                       seq_len, hidden, shared_inter);   // [860, 896] × [896, 2560]
ds_swiglu_direct(s_swiglu, s_gate, s_up, seq_len, shared_inter);
ds_linear_nobias_bf16(shared_mlp_out, s_swiglu, layer->shared_down_weight_bf16,
                       seq_len, shared_inter, hidden);   // [860, 2560] × [2560, 896]
```

**加速效果**：860 个 token 从 860 次 matvec 变为 3 次 sgemm，BLAS 效率提升 **50-100×**。

### 3.3 优化二：路由专家按 Expert ID 分组

路由专家的挑战是：不同 token 选择不同专家（top-6 of 64），不能简单批量处理。

**核心洞察**：860 个 token 中，平均每个专家被 ~860×6/64 ≈ 80 个 token 选中。把选中同一专家的 token 收集到一起，就能批量 sgemm。

```
v0.5（逐 token）:
  token 0 → expert 3, 17, 22, 41, 55, 60  → 6 次单 matvec
  token 1 → expert 7, 12, 22, 35, 48, 61  → 6 次单 matvec
  ...                                     ─────────────────
  总计 860 × 6 = 5160 次单 matvec

v0.9（按 expert 分组）:
  expert 3:  token [0, 15, 33, ...]  → 一次 sgemm [80, 896] × [896, 1280]
  expert 7:  token [1, 22, 47, ...]  → 一次 sgemm [80, 896] × [896, 1280]
  ...                               ────────────────────────────────────
  总计 ≤64 次 sgemm（大部分 ~80 token 的批量）
```

**代码实现**（`ds_moe_decoder.c`）：

```c
/* 统计每个 expert 被多少 token 选中 */
int *expert_count = calloc(n_experts, sizeof(int));
for (int s = 0; s < seq_len; s++) {
    int *tk_idx = all_top_indices + s * top_k;
    for (int k = 0; k < top_k; k++)
        expert_count[tk_idx[k]]++;
}

/* 为每个 expert 构建 token 索引列表 */
int **expert_tokens = malloc(n_experts * sizeof(int *));
for (int e = 0; e < n_experts; e++)
    expert_tokens[e] = malloc(expert_count[e] * sizeof(int));

/* 逐 expert 批量处理 */
for (int e = 0; e < n_experts; e++) {
    int n_tok = expert_count[e];
    if (n_tok == 0) continue;

    /* Gather: 收集选中此 expert 的所有 token 输入 */
    float *x_expert = malloc(n_tok * hidden * sizeof(float));
    for (int i = 0; i < n_tok; i++)
        memcpy(x_expert + i * hidden,
               x_norm + expert_tokens[e][i] * hidden,
               hidden * sizeof(float));

    /* 批量 sgemm：gate, up, swiglu, down */
    ds_linear_nobias_bf16(e_gate, x_expert, gate_weight,
                           n_tok, hidden, moe_inter);
    ds_linear_nobias_bf16(e_up, x_expert, up_weight,
                           n_tok, hidden, moe_inter);
    ds_swiglu_direct(e_swiglu, e_gate, e_up, n_tok, moe_inter);
    ds_linear_nobias_bf16(e_out, e_swiglu, down_weight,
                           n_tok, moe_inter, hidden);

    /* Scatter-add: 将结果加权写回对应 token 位置 */
    for (int i = 0; i < n_tok; i++) {
        int s = expert_tokens[e][i];
        float w = find_routing_weight(all_top_weights, s, top_k, e);
        for (int j = 0; j < hidden; j++)
            routed_mlp_out[s * hidden + j] += w * e_out[i * hidden + j];
    }
    free(x_expert); free(e_gate); free(e_up);
    free(e_swiglu); free(e_out);
}
```

> **Gather-Compute-Scatter 模式**：这是 MoE 批量推理的经典范式——先 Gather 收集输入、再 Compute 批量计算、最后 Scatter 散射结果。代价是额外的内存分配和拷贝，但换来的 sgemm 加速远超这些开销。

### 3.4 性能模型

```
v0.5: 860 × 6 × 3 × ~1.8ms/BF16_matvec ≈ 27s
v0.9: 共享 3 次 sgemm(~5ms) + 路由 ≤64 次 sgemm(~10ms) + Gather/Scatter(~2ms) ≈ 0.9s
加速比 ≈ 30×
```

### 3.5 注意力层的批量处理

除了 MoE，Prefill 的注意力层也从逐 token 变为批量矩阵乘法：
- QKV 投影：`[seq_len, hidden] × [hidden, 3×kv_dim]` → 一次 sgemm
- Attention：利用在线 softmax（O(1) 内存），不需要完整注意力矩阵
- Out 投影：`[seq_len, hidden] × [hidden, hidden]` → 一次 sgemm

## 4. Decode 优化：19s → 5.1s（3.7×）

Decode 阶段逐 token 生成，每个 token 独立执行 12 层 decoder 前向传播。优化目标是**减少单步延迟**。

### 4.1 Argmax LM Head（v0.8，8× 单步加速）

**问题**：每个 decode step 需要从 129280 个候选 token 中选最优。v0.5 用 sgemm 计算全部 logit，需要 631MB BF16→F32 转换。

**优化**：贪心解码不需要全部 logit，只需最大值。**边算边比较**（`ds_argmax_matvec_bf16`）：

```c
/* v0.9: argmax matvec —— 逐行 BF16×F32 点积，追踪最大值 */
float best_val = -1e30f;
int best_token = 0;
for (int row = 0; row < 129280; row++) {
    float dot = ds_bf16_dot_f32(x, lm_head_bf16 + row * 1280, 1280);
    if (dot > best_val) { best_val = dot; best_token = row; }
}
// 时间: ~8ms (vs v0.5 的 ~63ms，7.9× 加速)
```

### 4.2 Selective Repetition Penalty（v0.9）

只重算历史中出现过的 token（~100 个 vs 129280 个）：

```c
for (int h = 0; h < n_history; h++) {
    int tid = token_history[h];
    float logit = ds_bf16_dot_row(x, lm_head_bf16, tid, hidden);
    logit = (logit > 0) ? logit * rp : logit / rp;
    if (logit > best_val) { best_val = logit; best_token = tid; }
}
```

### 4.3 连续 Expert 块（P2，2.4× MoE 加速）

将 64 个 expert 的 gate_up_fused 权重分配在**一块连续内存**中，相邻 expert 共享 4KB 内存页：

```c
/* v0.9: 连续 expert 块 */
size_t per_expert = 2 * moe_inter * hidden * sizeof(uint16_t);
size_t block_size = n_experts * per_expert;
uint16_t *expert_block = malloc(block_size);  // 单次大块分配
for (int e = 0; e < n_experts; e++)
    layer->experts[e].gate_up_fused_bf16 = expert_block + e * per_expert;
```

**效果**：page fault 从 ~351ms/step 降至 ~121ms/step（**2.4× 减少**）。

### 4.4 madvise 预取

```c
/* 异步预读下一个 expert 的权重 */
if (k + 1 < top_k) {
    madvise(layer->experts[next_id].down_weight_bf16,
            hidden * moe_inter * 2, MADV_WILLNEED);
}
```

### 4.5 Fused Expert Forward

gate + up 投影融合为单次 matvec，输入向量只读一次：

```c
/* v0.9: 1 次 matvec (fused gate+up) */
ds_bf16_matvec(gate_up_buf, x, gate_up_fused_weight, hidden, 2*moe_inter);
memcpy(gate_buf, gate_up_buf, moe_inter * sizeof(float));
memcpy(up_buf, gate_up_buf + moe_inter, moe_inter * sizeof(float));
```

## 6. V3 R-SWA 优化效果

### 6.1 问题：KV Cache 线性增长

```
生成 1000 token: KV cache ≈ 5 MB
生成 10000 token: KV cache ≈ 512 MB  ← 爆内存！
```

### 6.2 R-SWA 解决方案

**Reference Sliding Window Attention** 将解码注意力分为两段：

```
KV Cache 布局:
[0..prefill_len-1]       [prefill_len..prefill_len+W-1]
├─ Reference 段 ────────┤├──── Sliding Window 段 ─────────┤
│ 视觉 token + prompt   ││ 最近 W=128 个 decode token     │
│ → 全局可见            ││ → 环形缓冲区，新覆盖旧         │
│ → 不参与状态转移      ││ → 固定大小                      │
└────────────────────────┘└───────────────────────────────┘
KV Cache 总大小 = prefill_len + W = 恒定值
```

### 6.3 C 实现要点

```c
/* ds_kernels.c: ds_rswa_attention_aligned() */
/* 两段在线 softmax */
for (int h = 0; h < n_heads; h++) {
    /* Reference 段: [0, prefill_len) — 视觉信息始终可见 */
    for (int i = 0; i < prefill_len; i++)
        qk_ref += Q[i] * K[i];

    /* Sliding window 段: [max(0, pos-W), pos) — 近期文本 */
    int win_start = (pos > window_size) ? pos - window_size : 0;
    for (int i = win_start; i < pos; i++)
        qk_win += Q[i] * K[i];

    /* 在线 softmax 合并两段 */
    ...
}
```

| 指标 | V3 (R-SWA) | V2 (标准) |
|------|-----------|----------|
| Decode 速度 | 44.0 tok/s | 44.4 tok/s |
| KV Cache (1000 tok) | prefill_len+128 ≈ 406 | 1000+ |
| KV Cache (10000 tok) | **恒定** 406 | 10000+ |
| 长文档内存 | ✅ 常数 | ❌ 线性增长 |

## 7. 内存带宽瓶颈分析

### 7.1 MoE Decode 每步权重读取量

| 权重 | 大小 (BF16) |
|------|------------|
| QKV + Out 投影 | ~12.7 MB/层 |
| 6 路由专家 | ~39.5 MB/层 |
| 2 共享专家 | ~26.4 MB/层 |
| **单层总计** | **≈ 78.6 MB** |
| **12 层 + LM head** | **≈ 943 + 331 MB** |

### 7.2 理论极限 vs 实测

```
M2 Pro 内存带宽: ~200 GB/s
冷缓存理论极限: 943 MB / 200 GB/s ≈ 4.7 ms/step ≈ 212 tok/s
实测: ~23 ms/step ≈ 44 tok/s
效率: ~20%
```

> 20% 效率的原因：BF16→F32 转换、线程同步、非连续内存访问、memory-bound 计算。

## 9. 优化建议与未来方向

### 9.1 短期可行（预计 2-4× decode 加速）

| 优化 | 原理 | 预期收益 | 实现难度 |
|------|------|---------|---------|
| **INT8 权重量化** | BF16→INT8，带宽减半 | 2× decode | 中 |
| **INT4 权重量化** | BF16→INT4，带宽降 4× | 3-4× decode | 高 |
| **Expert 量化混合** | 热门 expert BF16，冷门 INT4 | 2-3× decode | 高 |

### 9.2 中期探索

| 优化 | 原理 | 预期收益 |
|------|------|---------|
| **Speculative Decoding** | 小模型预测多 token，大模型验证 | 2-3× decode |
| **Shared + Routed Expert 并行** | 两者无依赖，可在不同线程执行 | 1.3× decode |
| **KV Cache 压缩** | 量化或蒸馏 KV cache | 长序列场景 |
| **FlashAttention (prefill)** | 减少注意力的内存读写 | prefill 加速 |

### 9.3 长期方向

- **Batched Decode**：多请求批量 decode，提升吞吐量
- **MoE Expert 缓存**：LRU 缓存热门 expert 的 F32 权重
- **异步 I/O**：decode 计算与 expert 权重预取重叠
- **动态线程分配**：小 matvec 用少线程，大 matvec 用多线程

## 小结

ds-ocr 从 v0.5（97s）到 v0.9（15s）的 6.5× 加速，每一倍都来自对瓶颈的精确测量和针对性优化：

1. **Encoding**：并行 N+1 workers → 5.5×（v0.5→v0.8）
2. **Prefill**：Batched MoE sgemm → 32×（v0.5→v0.8）
3. **Decode**：Argmax LM head + 连续 expert 块 + madvise → 3.7×（v0.5→v0.9）
4. **V3 R-SWA**：KV cache 恒定化 → 长文档无限制解析

**核心方法论**：
- **测量先行**：用 Profiler 精确定位瓶颈
- **Amdahl 定律**：优先优化占比最大的部分
- **硬件友好**：连续内存、批量计算、减少 page fault
- **精度优先**：BF16 路由确保与 Python 一致，优化不能牺牲正确性

带着这些知识，你可以在自己的推理引擎中应用同样的优化策略——**测量 → 分析 → 优化 → 验证**，循环迭代，直到接近硬件极限。
