## 1. 为什么用纯 C 写推理引擎？

### 1.1 Python 推理 vs C 推理

| | Python (PyTorch) | 纯 C 推理 |
|--|-------------------|-----------|
| **启动速度** | 慢（导入 PyTorch ~2s） | 瞬间 |
| **内存占用** | 大（PyTorch 运行时 ~1GB+） | 小（只有权重+缓冲区） |
| **依赖** | PyTorch, CUDA, Python... | 零依赖（只用系统 BLAS） |
| **移植性** | 依赖 Python 环境 | 任何有 C 编译器的平台 |
| **控制力** | 抽象层厚 | 直接控制每个操作 |
| **适合场景** | 研究/训练 | 生产部署/嵌入式 |

### 1.2 antirez 的设计哲学

qwen-asr（由 Redis 作者 antirez 开发）的设计哲学：
1. **零依赖**：不需要 Python、PyTorch、CUDA
2. **简洁**：每个函数做一件事，逻辑清晰
3. **高效**：BF16 零拷贝 + 平台优化内核
4. **可读**：代码即文档，从代码就能理解算法

ds-ocr 完全继承了这一哲学，并扩展到视觉和多模态领域。

## 2. Safetensors：权重的存储与加载

### 2.1 Safetensors 格式

Safetensors 是 HuggingFace 推出的模型权重格式，比 PyTorch 的 `.bin` 更安全、更快：

```
┌─────────────────────────────────────┐
│ 8 bytes: N = header JSON 长度       │
├─────────────────────────────────────┤
│ N bytes: header JSON                │
│  {"tensor_name": {                  │
│    "dtype": "BF16",                 │
│    "shape": [1280, 896],            │
│    "data_offsets": [start, end]     │
│  }, ...}                            │
├─────────────────────────────────────┤
│ 数据区域: 原始二进制 (BF16/F32/...)  │
│  tensor1 的数据 | tensor2 的数据 | ...│
└─────────────────────────────────────┘
```

### 2.2 零拷贝 mmap 加载

传统方式：打开文件 → read 到内存 → 解析  
mmap 方式：把文件映射到虚拟地址空间 → 直接用指针访问

**在 ds-ocr 中的实现**（`ds_safetensors.c`）：

```c
safetensors_file_t *safetensors_open(const char *path) {
    /* 1. 打开文件 */
    int fd = open(path, O_RDONLY);
    
    /* 2. 获取文件大小 */
    struct stat st;
    fstat(fd, &st);
    size_t file_size = st.st_size;
    
    /* 3. mmap: 将文件映射到内存 */
    void *data = mmap(NULL, file_size, PROT_READ, MAP_PRIVATE, fd, 0);
    
    /* 4. 解析 header JSON（前 8 bytes 是长度，后面是 JSON） */
    uint64_t header_len;
    memcpy(&header_len, data, 8);
    char *header_json = (char *)data + 8;
    
    /* 5. 解析 JSON，记录每个 tensor 的偏移和形状 */
    /* ... */
}
```

> **关键优势**：mmap 是**惰性加载**——只有实际被访问的页才会从磁盘读入。对于 3B 参数的模型，大部分专家权重可能根本不会被访问到（MoE 稀疏激活），节省大量内存。

### 2.3 多分片支持

大模型权重通常分成多个文件（`model-00001-of-00003.safetensors` 等）：

```c
typedef struct {
    safetensors_file_t *shards[8];  /* 最多 8 个分片 */
    int num_shards;
} multi_safetensors_t;

/* 统一查找：在所有分片中搜索 tensor */
const safetensor_t *multi_safetensors_find(const multi_safetensors_t *ms,
                                            const char *name,
                                            safetensors_file_t **out_sf);
```

### 2.4 BF16 权重的零拷贝访问

```c
/* 获取 BF16 数据的直接指针（不复制） */
uint16_t *safetensors_get_bf16_direct(const safetensors_file_t *sf,
                                       const safetensor_t *t) {
    return (uint16_t *)((char *)sf->data + sf->header_size + t->data_offset);
}

/* 获取 F32 数据（需要转换，会复制） */
float *safetensors_get_f32(const safetensors_file_t *sf,
                            const safetensor_t *t) {
    uint16_t *bf16 = safetensors_get_bf16_direct(sf, t);
    size_t n = safetensor_numel(t);
    float *f32 = (float *)malloc(n * sizeof(float));
    /* BF16 → F32 转换 */
    uint32_t *d = (uint32_t *)(void *)f32;
    for (size_t i = 0; i < n; i++)
        d[i] = ((uint32_t)bf16[i]) << 16;
    return f32;
}
```

> **策略**：解码器权重（占 95%+）用 BF16 零拷贝，编码器权重（较小）用 F32 预转换。因为解码器是逐 token 推理（计算密度低），BF16→F32 的逐行转换开销可忽略；而编码器是批量计算，预转换更高效。

## 3. BF16 精度：存储一半，计算全精度

### 3.1 什么是 BF16？

| 格式 | 符号位 | 指数位 | 尾数位 | 范围 | 精度 |
|------|--------|--------|--------|------|------|
| FP32 | 1 | 8 | 23 | ±3.4e38 | ~7位十进制 |
| FP16 | 1 | 5 | 10 | ±65504 | ~3位十进制 |
| **BF16** | 1 | 8 | 7 | ±3.4e38 | ~2位十进制 |

BF16 的巧妙之处：**保持 FP32 的指数范围，只减少尾数精度**。
- 和 FP32 范围相同（不会溢出）
- 精度只有 FP32 的 1/256（但对深度学习足够）
- 只需 FP32 一半的存储空间

### 3.2 BF16 ↔ FP32 转换

```
BF16 (16bit):  S EEEEEEEE MMMMMMM
FP32 (32bit):  S EEEEEEEE MMMMMMMMMMMMMMMMMMMMMMM

转换: FP32 = BF16 << 16  (高位对齐，低位补零)
```

**在代码中**：

```c
/* BF16 → FP32 */
uint32_t f32_bits = ((uint32_t)bf16_value) << 16;
float f32_value;
memcpy(&f32_value, &f32_bits, sizeof(float));

/* FP32 → BF16 (截断低16位) */
uint32_t f32_bits;
memcpy(&f32_bits, &f32_value, sizeof(uint32_t));
uint16_t bf16_value = (uint16_t)(f32_bits >> 16);
```

### 3.3 BF16 权重的矩阵乘法

ds-ocr 的核心操作：**用 BF16 权重和 F32 输入做矩阵乘法，输出 F32 结果**。

```c
/* y = x @ W_bf16^T  (x: F32, W: BF16, y: F32) */
void ds_linear_nobias_bf16(float *y, const float *x, const uint16_t *W_bf16,
                            int seq_len, int in_dim, int out_dim) {
    /* 内部实现：
     * 逐行遍历 W_bf16，每行：
     *   1. BF16 → F32 (on-the-fly)
     *   2. dot(x, row_f32) → y[row]
     */
}
```

> **为什么不预先全部转 F32？** 因为解码器权重巨大（几 GB），全转 F32 会翻倍内存。逐元素 on-the-fly 转换只需少量寄存器，不增加内存占用。

### 3.4 流式 Argmax：不计算全部 logits

对于 12 万词的词表，计算全部 logits（lm_head @ x）需要 129280 × 1280 = 1.65 亿次乘法。但贪心解码只需要 argmax——**最大值在哪就行**。

**流式 Argmax**（`ds_kernels.c` → `ds_argmax_matvec_bf16`）：

```c
int ds_argmax_matvec_bf16(const float *x, const uint16_t *W_bf16,
                           int in_dim, int out_dim) {
    int best_id = 0;
    float best_val = -1e30f;
    
    /* 逐行计算点积，同时追踪最大值 */
    for (int row = 0; row < out_dim; row++) {
        float dot = ds_dot_f32_impl(x, bf16_to_f32_row(row), in_dim);
        if (dot > best_val) {
            best_val = dot;
            best_id = row;
        }
    }
    return best_id;
    /* 不需要存储 129280 个 logits！ */
}
```

> **节省内存**：流式 argmax 不需要分配 129280 × 4 bytes = 512KB 的 logits 缓冲区。但对于需要重复惩罚等后处理的场景，仍需完整 logits。

## 4. 数学内核：从通用到平台优化

### 4.1 三级内核架构

ds-ocr 和 qwen-asr 都采用**三级内核架构**：

```
ds_kernels.h           ← 统一 API（平台无关）
    │
ds_kernels_impl.h      ← 编译时平台检测 + 宏分发
    │
    ├── ds_kernels_generic.c   ← 通用 C 实现（任何平台）
    ├── ds_kernels_neon.c      ← ARM NEON 优化（Apple Silicon）
    └── ds_kernels_avx.c       ← x86 AVX2+FMA 优化（Intel/AMD）
```

**编译时自动选择**（`ds_kernels_impl.h`）：

```c
#ifdef __ARM_NEON
    #define ds_bf16_matvec_fused_impl ds_bf16_matvec_fused_neon
    #define ds_dot_f32_impl ds_dot_f32_neon
#elif defined(__AVX2__) && defined(__FMA__)
    #define ds_bf16_matvec_fused_impl ds_bf16_matvec_fused_avx
    #define ds_dot_f32_impl ds_dot_f32_avx
#else
    #define ds_bf16_matvec_fused_impl ds_bf16_matvec_fused_generic
    #define ds_dot_f32_impl ds_dot_f32_generic
#endif
```

### 4.2 以 dot product 为例

**通用实现**：

```c
/* ds_kernels_generic.c */
float ds_dot_f32_generic(const float *a, const float *b, int n) {
    float sum = 0.0f;
    for (int i = 0; i < n; i++) sum += a[i] * b[i];
    return sum;
}
```

**NEON 实现**（4× 并行）：

```c
/* ds_kernels_neon.c */
float ds_dot_f32_neon(const float *a, const float *b, int n) {
    float32x4_t sum_vec = vdupq_n_f32(0.0f);
    int i = 0;
    for (; i + 3 < n; i += 4) {
        float32x4_t a_vec = vld1q_f32(a + i);
        float32x4_t b_vec = vld1q_f32(b + i);
        sum_vec = vfmaq_f32(sum_vec, a_vec, b_vec);  /* FMA: sum += a*b */
    }
    /* 水平求和 + 处理剩余元素 */
    float sum = vaddvq_f32(sum_vec);
    for (; i < n; i++) sum += a[i] * b[i];
    return sum;
}
```

**AVX2 实现**（8× 并行）：

```c
/* ds_kernels_avx.c */
float ds_dot_f32_avx(const float *a, const float *b, int n) {
    __m256 sum_vec = _mm256_setzero_ps();
    int i = 0;
    for (; i + 7 < n; i += 8) {
        __m256 a_vec = _mm256_loadu_ps(a + i);
        __m256 b_vec = _mm256_loadu_ps(b + i);
        sum_vec = _mm256_fmadd_ps(a_vec, b_vec, sum_vec);
    }
    /* 水平求和 + 剩余处理 */
}
```

### 4.3 BLAS 加速的矩阵乘法

对于大矩阵乘法（prefill 阶段），ds-ocr 和 qwen-asr 都调用系统 BLAS：

```c
/* macOS: Accelerate Framework (cblas_sgemm) */
/* Linux: OpenBLAS (cblas_sgemm) */

/* C = A @ B^T: A[seq,K], B[N,K], C[seq,N] */
void ds_matmul_t(float *C, const float *A, const float *B, int M, int K, int N) {
    cblas_sgemm(CblasRowMajor, CblasNoTrans, CblasTrans,
                M, N, K, 1.0f, A, K, B, K, 0.0f, C, N);
}
```

> **为什么 BLAS 更快？** BLAS 库针对特定 CPU 微架构做了极致优化（cache 友好分块、指令级并行、寄存器分配），比手写循环快 5-10×。

### 4.4 BF16 融合矩阵向量乘

单 token 解码时，核心操作是 `y = W_bf16^T @ x`（矩阵×向量）。ds-ocr 提供了高度优化的融合实现：

```c
/* ds_kernels_neon.c — NEON 优化的 BF16 matvec */
void ds_bf16_matvec_fused_neon(float *y, const float *x, const uint16_t *W_bf16,
                                 const float *bias, int in_dim, int out_dim) {
    /* 同时做 BF16→F32 转换 + 点积 + 加 bias
     * 利用 NEON 的 BF16 指令直接在 BF16 上做乘法
     * 避免了先转 F32 再乘的额外内存访问 */
}
```

## 5. KV 缓存：自回归推理的加速器

### 5.1 为什么需要 KV 缓存？

自回归解码时，每个新 token 需要和**所有之前的 token** 做注意力计算。如果没有缓存：

```
Step 1: 计算 K₁,V₁ → Q₁·K₁ → 输出
Step 2: 计算 K₁,K₂,V₁,V₂ → Q₂·[K₁,K₂] → 输出  (重复计算 K₁,V₁!)
Step 3: 计算 K₁,K₂,K₃,V₁,V₂,V₃ → Q₃·[K₁,K₂,K₃] → 输出  (重复更严重!)
```

KV 缓存：**把已计算的 K、V 存起来，每步只算新 token 的 K、V**：

```
Step 1: 算 K₁,V₁ → 存入缓存 → Q₁·K₁ → 输出
Step 2: 算 K₂,V₂ → 存入缓存 → Q₂·[缓存K₁,K₂] → 输出  (不重复计算!)
Step 3: 算 K₃,V₃ → 存入缓存 → Q₃·[缓存K₁,K₂,K₃] → 输出
```

> **节省**：Step t 的注意力计算从 O(t²d) 降到 O(td)（只需计算新 Q 与缓存 KV 的点积）。

### 5.2 KV 缓存的数据结构

**ds-ocr 中的 KV 缓存**（`ds_ocr.h`）：

```c
/* KV 缓存: [layers × max_seq × kv_dim] */
float *kv_cache_k;  /* Key 缓存 */
float *kv_cache_v;  /* Value 缓存 */
int kv_cache_len;   /* 当前已缓存的位置数 */
int kv_cache_max;   /* 最大缓存容量 */
```

**内存布局**：`kv_cache_k[layer][position][kv_dim]`

```c
/* 分配 KV 缓存 */
int max_seq = 4096;
int kv_dim = n_kv_heads * head_dim;  /* 10 * 128 = 1280 */
size_t kv_size = (size_t)12 * max_seq * kv_dim * sizeof(float);
/* 12 层 × 4096 位置 × 1280 维 × 4 bytes = ~240MB */
ctx->kv_cache_k = (float *)calloc(1, kv_size);
ctx->kv_cache_v = (float *)calloc(1, kv_size);
```

### 5.3 解码时存取 KV 缓存

```c
/* 写入：新 token 的 K、V 存入缓存 */
int cache_offset = ctx->kv_cache_len;
float *cache_k = ctx->kv_cache_k + (size_t)layer_idx * kv_cache_max * kv_dim
                  + cache_offset * kv_dim;
float *cache_v = ctx->kv_cache_v + (size_t)layer_idx * kv_cache_max * kv_dim
                  + cache_offset * kv_dim;
memcpy(cache_k, k, kv_dim * sizeof(float));
memcpy(cache_v, v, kv_dim * sizeof(float));

/* 读取：因果注意力时，使用缓存中的全部 K、V */
float *full_k = ctx->kv_cache_k + (size_t)layer_idx * kv_cache_max * kv_dim;
float *full_v = ctx->kv_cache_v + (size_t)layer_idx * kv_cache_max * kv_dim;
ds_causal_attention(attn_out, q, full_k, full_v,
                    1, cache_offset + 1, ...);
```

### 5.4 GQA 对 KV 缓存的影响

| 架构 | Q头 | KV头 | KV缓存大小 |
|------|-----|------|-----------|
| MHA (ds-ocr解码器) | 10 | 10 | 10 × 128 = 1280/position |
| GQA (qwen-asr解码器) | 16 | 8 | 8 × 128 = 1024/position |
| GQA (ds-ocr V2编码器) | 14 | 2 | 2 × 64 = 128/position |

GQA 通过共享 KV 头，直接减少了 KV 缓存的内存占用。qwen-asr 使用 GQA 后，KV 缓存减少 50%。

### 5.5 qwen-asr 的动态 KV 缓存扩展

qwen-asr 支持动态扩展 KV 缓存：

```c
static int kv_cache_grow(qwen_ctx_t *ctx, int required) {
    if (required <= ctx->kv_cache_max) return 0;
    int new_max = ctx->kv_cache_max;
    while (new_max < required) new_max *= 2;  /* 倍增策略 */
    
    /* 重新分配 + 拷贝旧数据 */
    float *new_k = (float *)calloc(1, total_size);
    float *new_v = (float *)calloc(1, total_size);
    for (int l = 0; l < layers; l++) {
        memcpy(new_k + l * new_stride, old_k + l * old_stride, copy_size);
        memcpy(new_v + l * new_stride, old_v + l * old_stride, copy_size);
    }
    /* ... */
}
```

## 6. Prefill vs Decode：两种推理模式

### 6.1 两种模式的区别

| | Prefill | Decode |
|--|---------|--------|
| **输入** | 多个 token（整个 prompt） | 单个 token |
| **计算方式** | 批量矩阵乘法 (sgemm) | 矩阵×向量 (matvec) |
| **注意力** | 因果掩码 + 批量 | 因果掩码 + KV缓存 |
| **KV缓存** | 全部写入 | 只写1个位置 |
| **瓶颈** | 计算（sgemm 快） | 内存带宽（matvec 慢） |
| **速度** | 快（并行） | 慢（串行） |

### 6.2 Prefill 的实现

**ds-ocr 中**（`ds_moe_decoder.c` → `ds_decoder_prefill`）：

```c
void ds_decoder_prefill(ds_ctx_t *ctx, const float *input_embeds, int seq_len) {
    /* 分配批量缓冲区 */
    float *x = (float *)malloc(seq_len * hidden * sizeof(float));
    float *Q = (float *)malloc(seq_len * q_dim * sizeof(float));
    /* ... */
    
    for (int l = 0; l < cfg->dec_layers; l++) {
        /* 批量 RMSNorm */
        ds_rms_norm(x_norm, x, layer->input_norm, seq_len, hidden, eps);
        
        /* 批量 QKV 投影（sgemm） */
        ds_linear_nobias_bf16(Q, x_norm, layer->wq_weight_bf16, seq_len, hidden, q_dim);
        
        /* 批量注意力 */
        ds_causal_attention(attn_out, Q, full_k, full_v, seq_len, total_seq, ...);
        
        /* MoE：批量 gate + 逐 token 专家 */
        ds_linear_nobias_bf16(gate_scores, x_norm, layer->gate_weight_bf16,
                               seq_len, hidden, n_experts);  /* 批量路由 */
        for (int s = 0; s < seq_len; s++) {
            /* 逐 token 做专家前向（和 decode 相同） */
        }
    }
}
```

### 6.3 Decode 的实现

**ds-ocr 中**（`ds_moe_decoder.c` → `ds_decoder_forward`）：

```c
int ds_decoder_forward(ds_ctx_t *ctx, const float *input_embed) {
    /* 使用预分配的单 token 缓冲区 */
    float *x = ctx->dec_x;       /* [1280] */
    float *q = ctx->dec_q;       /* [1280] */
    float *k = ctx->dec_k;       /* [1280] */
    
    for (int l = 0; l < cfg->dec_layers; l++) {
        /* 单 token 前向 */
        decoder_layer_forward(ctx, x, out, l, ctx->kv_cache_len);
        /* 指针交换，避免 memcpy */
        float *tmp = x; x = out; out = tmp;
    }
    
    /* 最终归一化 + argmax */
    ds_rms_norm(x, x, dec->norm, 1, hidden, eps);
    return ds_argmax_matvec_bf16(x, dec->lm_head_bf16, hidden, vocab);
}
```

### 6.4 指针交换优化

注意 `decoder_layer_forward` 中的指针交换：

```c
/* 不做 memcpy，而是交换指针 */
float *tmp = x; x = out; out = tmp;
```

这避免了每次层输出时 1280×4 = 5KB 的内存拷贝。12 层下来节省 60KB 拷贝/步，生成 1000 个 token 就是 60MB 的节省。

## 7. 采样与解码：从 logits 到文字

### 7.1 贪心解码（Greedy）

最简单的解码方式：每次选概率最大的 token。

```c
/* ds_argmax_matvec_bf16: 流式找到最大 logit 的 token ID */
int token = ds_argmax_matvec_bf16(x, dec->lm_head_bf16, hidden, vocab);
```

### 7.2 温度采样（Temperature Sampling）

当 `temperature > 0` 时，引入随机性：

```c
/* 1. 对 logits 应用温度 */
for (int i = 0; i < vocab; i++)
    logits[i] = expf((logits[i] - max_logit) / temperature);

/* 2. Softmax 归一化 */
double sum_exp = 0.0;
for (int i = 0; i < vocab; i++) sum_exp += logits[i];
for (int i = 0; i < vocab; i++) logits[i] /= sum_exp;

/* 3. 加权随机采样 */
float r = (float)rand() / RAND_MAX;
float cum = 0.0f;
for (int i = 0; i < vocab; i++) {
    cum += logits[i];
    if (cum >= r) { token = i; break; }
}
```

> **温度越高**，分布越平坦，生成越随机；**温度越低**，越接近贪心。

### 7.3 重复惩罚（Repetition Penalty）

防止模型重复输出相同内容：

```c
float rp = ctx->repeat_penalty;  /* 1.0 = 无惩罚, 1.1~1.5 常用 */
for (int i = 0; i < token_history_len; i++) {
    int tid = ctx->token_history[i];
    if (logits[tid] > 0) logits[tid] /= rp;   /* 正 logit 缩小 */
    else logits[tid] *= rp;                     /* 负 logit 放大（更负） */
}
```

### 7.4 N-gram 阻断

防止重复输出相同的 n-gram（连续 n 个 token 的组合）：

```c
int ngram_n = ctx->no_repeat_ngram_size;  /* 默认 35 */
if (ngram_n > 0 && token_history_len >= ngram_n - 1) {
    int prefix_len = ngram_n - 1;
    /* 检查历史中是否有匹配的 n-gram 前缀 */
    for (int i = 0; i <= hist_len - prefix_len - 1; i++) {
        int match = 1;
        for (int j = 0; j < prefix_len; j++) {
            if (hist[i+j] != hist[hist_len-prefix_len+j]) { match = 0; break; }
        }
        if (match) {
            /* 禁止完成该 n-gram 的 token */
            logits[hist[i + prefix_len]] = -1e30f;
        }
    }
}
```

### 7.5 Token 解码：从 ID 到文字

BPE tokenizer 把 token ID 解码为 UTF-8 文本：

```c
/* ds_tokenizer.c */
const char *ds_tokenizer_decode(const ds_tokenizer_t *tok, int token_id) {
    /* 查表：id_to_text[token_id] */
    return tok->id_to_text[token_id];
}
```

## 8. 线程池与并行计算

### 8.1 为什么需要线程池？

解码器每步要做大量矩阵×向量运算，每个操作只占 1 个 CPU 核心。多核 CPU 可以**并行处理多个操作**。

### 8.2 ds-ocr 的线程池

```c
/* ds_kernels.c */
void ds_set_threads(int n);  /* 设置线程数（默认: CPU 核数） */
```

**典型使用场景**：

1. **批量 linear**：将 `seq_len × out_dim` 的矩阵乘法按行分给多个线程
2. **QKV 融合**：3 个矩阵乘法（Q、K、V）并行执行
3. **多 crop 并行**：ds-ocr 多裁剪编码时，每个 crop 独立编码

**ds-ocr 多 crop 并行**（`ds_ocr.c`）：

```c
/* 启动并行编码 */
for (int i = 0; i < n_crops; i++) {
    tasks[i].ctx = ctx;
    tasks[i].crop = crops[i];
    pthread_create(&threads[i], NULL, crop_worker, &tasks[i]);
}

/* 等待所有 crop 完成 */
for (int i = 0; i < n_crops; i++) {
    pthread_join(threads[i], NULL);
}
```

### 8.3 qwen-asr 的分块编码

qwen-asr 的音频编码器将长音频分块处理：

```c
int n_chunks = (mel_frames + chunk_size - 1) / chunk_size;
/* 每个 chunk 独立通过 Conv2D stem + Transformer */
for (int c = 0; c < n_chunks; c++) {
    /* 处理 chunk c... */
}
```

## 9. 两个项目的代码对比

### 9.1 架构对比

| | ds-ocr | qwen-asr |
|--|--------|----------|
| **输入** | 图像 (RGB pixels) | 音频 (Mel spectrogram) |
| **视觉/音频编码器** | SAM ViT-B / DeepEncoder V2 | Conv2D stem + Transformer |
| **投影层** | Projector (896→1280 / 2048→1280) | proj1+proj2 (d_model→output_dim) |
| **解码器** | MoE (64 experts + 2 shared) | Dense (SwiGLU) |
| **注意力** | MHA (解码器) + GQA (V2编码器) | GQA (解码器) + MHA (编码器) |
| **位置编码** | RoPE + 2D PE + 相对位置 | RoPE + 正弦PE |
| **权重精度** | BF16 (解码器) + F32 (编码器) | BF16→F32预转换 (编码器) + BF16 (解码器) |

### 9.2 代码结构对比

```
ds-ocr/                          qwen-asr/
├── ds_ocr.h/c          ←→      ├── qwen_asr.h/c         (总协调)
├── ds_visual_tokenizer  ←→      ├── qwen_asr_audio        (音频预处理)
├── ds_deep_encoder      ←→      ├── qwen_asr_encoder      (编码器)
├── ds_moe_decoder       ←→      ├── qwen_asr_decoder      (解码器)
├── ds_kernels*          ←→      ├── qwen_asr_kernels*     (数学内核)
├── ds_safetensors       ←→      ├── qwen_asr_safetensors  (权重加载)
├── ds_tokenizer         ←→      ├── qwen_asr_tokenizer    (BPE分词)
├── ds_image             ←→      (无: 音频无图像加载)
└── main.c               ←→      └── main.c               (CLI入口)
```

### 9.3 核心差异：MoE vs Dense

ds-ocr 解码器使用 MoE，qwen-asr 使用 Dense FFN。这是两者最大的架构差异：

**qwen-asr Dense 解码器层**：
```c
/* 简单直接的 SwiGLU FFN */
qwen_linear_nobias_bf16(gate_buf, x_norm, l->gate_up_fused_bf16, ...);
qwen_swiglu_multiply(gate_buf, gate_buf, 1, intermediate);
qwen_linear_nobias_bf16(ffn_out, gate_buf, l->down_weight_bf16, ...);
```

**ds-ocr MoE 解码器层**：
```c
/* 复杂的路由 + 多专家 + 共享专家 */
ds_moe_router_bf16(scores, x_norm, layer->gate_weight_bf16, ...);
ds_moe_top_k(top_indices, top_weights, scores, 64, 6);
for (int k = 0; k < 6; k++) {
    ds_expert_forward(expert_outputs + k*hidden, x, ...);
}
ds_expert_combine(output, expert_outputs, top_indices, top_weights, ...);
/* + 共享专家 */
```

## 小结

你现在理解了纯 C 推理引擎的实现细节：

1. **Safetensors + mmap**：零拷贝权重加载
2. **BF16**：存储减半，计算全精度，on-the-fly 转换
3. **三级内核**：generic → NEON/AVX → BLAS，编译时自动选择
4. **KV 缓存**：避免重复计算，GQA 减少缓存大小
5. **Prefill vs Decode**：批量 vs 单 token 两种模式
6. **采样**：贪心/温度/重复惩罚/n-gram 阻断
7. **线程池**：并行矩阵乘法 + 多 crop 并行编码
